## V3-1 Shadow 토큰

### 프롬프트 원문
> 아래 조건으로 워킹버디 shadow 토큰을 정의해줘.
>
> [피그마에서 확인된 실제 shadow값 - 이것만]
> - card shadow: 0px 0px 16px rgba(0,0,0,0.08)
> - nav shadow: 0px 0px 8px rgba(0,0,0,0.08)
> - glow-brand: 0px 0px 8px rgba(120,144,238,0.42)
>
> [절대 원칙]
> - 위 3개만 토큰화
> - 단계 확장 없음 (elevation-1~8 같은 시스템 X)

### 핵심 조건 / V1과 달라진 점
- V1은 Material Design 엘리베이션 시스템 8단계를 선제 생성
- V3는 피그마 실사용 3개만, 시스템 확장 없음을 명시

### 결과에서 달라진 점
| | V1 | V3 |
|---|---|---|
| 토큰 수 | 8개 (elevation 0~5 + glow 2종) | **3개** |
| 미사용 포함 | elevation 0, 1, 2, 5 | 없음 |

## V3-2 Bold(700) 처리

### 프롬프트 원문
> V2 결과물에서 Bold(700) 관련 항목을 수정해줘.
>
> [결정: 옵션 B — SemiBold 근사]
> 이유: V2에서 "Bold는 피그마 전체 미사용"으로 판단했고,
> 예외 하나를 위해 토큰을 추가하는 것보다
> 시스템 일관성을 유지하는 것이 우선.

### 핵심 조건 / V2와 달라진 점
- V2: 피그마 원본 Bold(700) 발견 → 미결로 남김
- V3: 예외 1개를 위한 토큰 추가보다 **시스템 일관성 우선** 원칙 적용

### 결과에서 달라진 점
| | V2 (피그마 원본) | V3 (확정) |
|---|---|---|
| TopBar 로고 weight | Bold(700) | `weight-semibold` (600) |
| 타이포 굵기 토큰 | 3가지 (+ 미결) | 3가지 (확정, 변경 없음) |

## V3-3 Nav Bottom 34px

### 프롬프트 원문
> 워킹버디 Navigation bottom의 padding-bottom 34px 처리 방식을 결정해줘.
>
> [결정: 옵션 A — 예외 고정값으로 명시]
> 이유: 34px는 iOS safe area 하단 여백으로,
> 디자인 의도가 아닌 디바이스 환경 대응값.
> spacing 시스템에 포함시키는 것이 오히려 부자연스러움.

### 핵심 조건 / V2와 달라진 점
- V2: spacing 토큰 범위 밖이라는 사실만 인식, 처리 방법 미결
- V3: 34px의 **성격(환경값 vs 디자인값)** 을 판단 기준으로 삼아 결정

### 결과에서 달라진 점
| | V2 | V3 |
|---|---|---|
| spacing 토큰 수 | 4개 | 4개 (변경 없음) |
| 34px 처리 | 미결 | 고정값으로 명시, 토큰 미편입 확정 |
| 코드 처리 방식 | 미정 | `padding-bottom: 34px` 직접 명시 |
