# V2 프롬프트 전체 기록

V1 문제점 인식 후 "단계별 제어" 방식으로 전환한 두 번째 시도.  
컬러 → 타이포그래피 → 스페이싱 → 레디어스 → 컴포넌트 순서로 영역별 분리 진행.

---

## 방법론 전환 (V1 → V2)

| 항목 | V1 | V2 |
|---|---|---|
| 접근 방식 | 전체 위임 ("알아서 재정립해줘") | 영역별 분리 + 단계별 확인 |
| 확장 기준 | Material Design 등 외부 시스템 참고 허용 | 피그마 실사용값만, 외부 확장 금지 |
| 진행 순서 | 한 번에 전체 생성 | Foundation 먼저, 컴포넌트는 마지막 |
| 중간 결정 | 없음 | 옵션 제시 → 선택 후 진행 (스페이싱) |

## V2-1 컬러

### 프롬프트 원문
> 아래 조건으로 워킹버디 Figma 파일의 컬러 토큰만 다시 정의해줘.
>
> [절대 원칙]
> - 피그마 화면에 실제로 등장한 컬러값만 포함한다
> - 피그마에 없는 컬러는 추가하지 않는다
> - Material Design 등 외부 시스템 기준으로 확장하지 않는다
>
> [피그마에서 확인된 실제 사용 컬러 - 이것만 사용]
> `#0077CE` / `#43A6EF` / `#FFFFFF` / `#F4F4F4` / `#EDEDED` / `#BBBBBB` / `#222222`
>
> [구조]
> Primitive 먼저: 위 7개 컬러를 이름 붙여서 나열 (확장 없이 7개만)
> Semantic 다음: Primitive 참조, 실제 UI 패턴 용도만 — bg / text / brand / border / disabled

### V1과 달라진 핵심 조건
- 피그마 실사용 7개만 → **확장 금지** 명시
- Semantic 범위를 용도 카테고리로 사전 지정 (bg / text / brand / border / disabled)
- 상태색(success / warning / error) 사전 차단

### 결과에서 달라진 점
| | V1 | V2 |
|---|---|---|
| Primitive 수 | 32개 (blue 11단계 + gray 12단계 + 상태색 9개) | **7개** |
| Semantic 수 | 14개 | **10개** |
| 상태색 포함 | 포함 | 미포함 (피그마에 없음) |
| **합계** | **46개** | **17개 (-63%)** |

## V2-2 타이포그래피

### 프롬프트 원문
> 아래 조건으로 워킹버디 타이포그래피 토큰만 다시 정의해줘.
>
> [절대 원칙]
> - 피그마 화면에서 실제로 사용된 텍스트 스타일만 포함한다
> - 피그마에 없는 스케일은 추가하지 않는다
>
> [피그마에서 확인된 실제 사용값]
> 사이즈: 11px, 12px, 14px, 16px, 18px, 72px (6단계)
> 굵기: Regular(400), Medium(500), SemiBold(600) — Bold 없음
>
> [구성해야 할 textStyles - 7종만]
> display(72) / title(18) / subtitle(16) / body-strong(14/SemiBold) / body(14/Regular) / caption(12) / label(11)
>
> line-height는 피그마 확인값만, 없으면 "-"로 표기

### V1과 달라진 핵심 조건
- textStyles 목록을 사전에 7종으로 명시 → AI가 임의 추가 불가
- Bold(700) 명시적 제외
- line-height 추측 금지 — 피그마 확인값만

### 결과에서 달라진 점
| | V1 | V2 |
|---|---|---|
| 사이즈 단계 | 11단계 (10~72px) | **6단계** |
| textStyles 수 | 13종 | **7종** |
| 굵기 종류 | 4가지 (Bold 포함) | **3가지** |
| line-height 종류 | 5가지 | **1가지 (1.4 고정 확인)** |

**추가 발견:** 모든 letter-spacing이 `-0.025em` 단일 규칙으로 통일되어 있음을 확인.  
**미결 이슈:** TopBar 로고 텍스트가 Bold(700)를 사용 — V2 타이포 정책과 충돌. V3에서 결정 필요.

## V2-3 스페이싱

### 프롬프트 원문
> 아래 조건으로 워킹버디 스페이싱 토큰을 다시 정의해줘.
>
> [상황 설명]
> 피그마에서 확인된 실제 여백값: 8px, 10px, 12px, 13px, 16px, 20px, 22px
> V1에서 4px 그리드로 생성했더니 10px, 22px 같은 중간값이 표현 안 되는 문제가 있었음.
>
> [두 가지 옵션을 제안해줘]
> 옵션 A: 피그마 실제 사용값 그대로 토큰화 (7개)
> 옵션 B: 4px 그리드 기반으로 근사화 — 조정 시 영향 컴포넌트 명시
>
> 각 옵션의 장단점 1~2줄로 설명. 내가 선택한 후 진행.

### V1과 달라진 핵심 조건
- 문제를 먼저 진단하고 옵션을 제시하는 구조 (단순 재정의 → 의사결정 포함)
- 조정 시 영향 범위를 명시 요청 → 결과 예측 가능하게

### 선택 및 결과
**선택: 옵션 B** (4px 그리드 근사화)  
→ 디자인 시스템을 기준으로 피그마를 맞추는 방향

| 조정 내용 | 원본 → 조정 | 영향 위치 |
|---|---|---|
| `10px → 12px` | +2px | 카드 섹션 래퍼 상하 여백 |
| `13px → 12px` | -1px | 상단바 아이콘 그룹 gap |
| `22px → 20px` | -2px | 버디 보내기 카드 레이블 위치 |

| | V1 | V2 |
|---|---|---|
| 토큰 수 | 16단계 | **4개 (-75%)** |
| 최댓값 | 96px | **20px** |
| 그리드 준수 | 불규칙 포함 | **4px 완전 준수** |

## V2-4 보더 레디어스

### 프롬프트 원문
> 아래 조건으로 워킹버디 보더 레디어스 토큰을 다시 정의해줘.
>
> [절대 원칙]
> - 피그마에서 실제 사용된 값만 포함한다
>
> [피그마에서 확인된 실제 사용값]
> 10px(small card) / 12px(card) / 9999px(pill/button)
>
> 3개 토큰만 정의. none(0)이나 미사용 단계는 포함하지 말 것.

### V1과 달라진 핵심 조건
- 미사용 단계 제외를 **명시** ("none이나 미사용 단계는 포함하지 말 것")
- 토큰 수를 3개로 사전 확정

### 결과에서 달라진 점
| | V1 | V2 |
|---|---|---|
| 토큰 수 | 8단계 (none~2xl~full) | **3개 (-63%)** |
| 미사용 포함 | xs(4), sm(8), xl(16), 2xl(24), none | **없음** |

## V2-5 컴포넌트

### 프롬프트 원문
> 아래 조건으로 워킹버디 컴포넌트 목록을 다시 정의해줘.
>
> [절대 원칙]
> - 피그마 화면 또는 컴포넌트 섹션에 실제로 존재하는 것만 포함한다
> - 피그마에 없는 컴포넌트는 만들지 않는다 (Badge, Input 제외)
>
> [피그마에서 확인된 실제 컴포넌트]
> Button(full-width) / Button(circle) / Card / ProgressBar /
> Checkbox / Navigation(bottom) / TopBar / Logo
>
> 구현 코드 X. 스펙만 정의:
> 컴포넌트명 / Variants(피그마 확인된 것만) / 사용 토큰 / 실제 크기·색상값

### V1과 달라진 핵심 조건
- 구현 코드 없이 **스펙 먼저** — 확인 후 구현
- 컴포넌트 목록을 사전에 8개로 확정
- Badge / Input 명시적 제외

### 결과에서 달라진 점
| | V1 | V2 |
|---|---|---|
| 구현 컴포넌트 | 4개 (Button, Card, ProgressBar, Badge) | **스펙 8개** |
| 미존재 컴포넌트 포함 | Badge, Input 포함 | **없음** |
| 피그마 존재 컴포넌트 누락 | Checkbox, Nav, TopBar, Logo 누락 | **전체 포함** |

**추가 발견:**
- Card가 3가지 형태(default / action-brand / action-surface)로 세분화됨
- Shadow값 3종이 아직 토큰화되지 않은 것 확인 → V3 미결 사항 등록
